how this works

every vote is a signed nostr event. our app is just one view of the public ledger. cloudflare is a cache. anyone can audit.
vote flow: iOS client signs vote with a per-device nostr keypair (BIP-340 Schnorr) ├─→ publishes signed event to 7 public nostr relays (the ledger) └─→ POSTs to cloudflare worker → D1 rollup (the cache) read path: this page → cloudflare worker → D1 → CDN-cached JSON (fast) auditors → any nostr relay → recompute from events (true) if the two ever disagree, the public ledger wins. the cache is opinion; the relays are the record.

verify the math

we don't ask you to trust us. here's exactly how to recompute today's leaderboard yourself from the public nostr ledger. if your numbers don't match ours, we're lying. call us out.
protocol: nostr NIP-01, kind 30078 (parameterized replaceable) filter: {"kinds":[30078], "#d":["blame.today:dev:vote:<YYYY-MM-DD>"]} (pre-launch namespace; cuts over to "blame.today:v1:vote:..." at launch) relays we publish to: relay.damus.io nos.lol relay.nostr.band relay.snort.social relay.primal.net offchain.pub nostr.mom steps: 1. connect to any relay above via WebSocket 2. send the filter (see /verify?day=<day> for the exact REQ) 3. for each EVENT, parse content as JSON → (pubkey, target) 4. group by target. unique pubkey count per target = real_count 5. compare to /leaderboard?day=<day> machine-readable: /verify?day=<today>

why this matters

blame.today started as a silly idea i'd had for years. when i tried to actually build it for free, an architecture happened — and once i looked at what the app would display, transparency mattered. the whole thing accidentally turned into a writeup of a reusable pattern: cache is opinion, protocol is record.

that same pattern would work for town meetings, HOA votes, public-spending dashboards, contractor reviews — anywhere the org publishing the data has incentive to bend it.

the writeup splits across two pages: why this matters → (motivation, values, posture — what we're trying to be, what we're not, the property the architecture is defending) and how this works → (math, architecture, mechanisms — platforms, seeding formula, moderation pipeline, attestation plan, user-flow logic).

why includes: what happens if you DDoS the worker, what happens if cloudflare goes down, what happens if i decide to start selling blames, and where else this pattern fits. how includes: the seeding math, the platform tradeoffs, what attestation does and doesn't promise.
