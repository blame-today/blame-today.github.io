what happens when things break

the interesting test of an architecture isn't how it behaves when everything works. it's how gracefully it fails. four scenarios — none of them break the property.

what if you DDoS our cloudflare worker?

cloudflare's anycast edge absorbs L3/L4 floods automatically — that's free, planetary-scale, and applies to every plan including ours. the real risk is exhausting our 100k req/day worker quota. defense in layers:

  • edge caching on /leaderboard?day=X means 99% of reads never invoke the worker
  • rate-limit rule on the worker endpoints — abusive IPs get bounced at the edge, not against our quota
  • bot fight mode trims obvious scripts
  • param validation rejects cache-busting query strings before they hit the database

worst case if all of that fails: the worker returns 429 until UTC midnight, the iOS app shows the last-cached board (it's local-first), and the nostr relays keep accepting real votes regardless. the ledger doesn't depend on us.

with a $5/mo hard cap on the worker, the worst-case financial outcome is $5 burned. an attacker rents a botnet costing more than that. the cost of attacking us is bounded; their incentive is too.

what if cloudflare goes down?

the same cache logic runs on AWS Lambda, Azure Functions, GCP Cloud Run, Deno Deploy. clients try the next provider. the relays kept running through the entire CF outage. eventual consistency does the rest.

today we're only on cloudflare. multi-provider rollout is in the release plan — but the protocol is platform-agnostic, so switching deployment targets is a worker rewrite, not a data migration.

what if blame.today the company gets sued or shut down?

the data isn't ours. the worker is a view. the canonical record is on 7 public nostr relays operated by independent parties, none of them named "blame.today inc." even hostile takedown of every relay we publish to is recoverable: any new relay accepts the same protocol, and clients can reconfigure. the protocol outlives the company.

what if we quietly miscount?

we publish the filter, the relay list, and the rules. anyone with a websocket can recompute today's leaderboard themselves. if our numbers diverge from the relays', we're lying — and it's detectable in seconds, not months. we cannot lie undetectably. that's the entire point.

where else this pattern fits

the cargo we're carrying is: cache is opinion, protocol is record. anywhere that property would matter:

  • public votes that need to stay verifiable — town meetings, HOA decisions, parent councils, community moderation
  • reputation systems that shouldn't be locked to one company — contractor reviews, restaurant trust, content authenticity
  • disclosure registries — lobbying contacts, public-spending dashboards, election counts, regulatory filings
  • accountability tools in general — anywhere "the org publishing the data" has incentive to bend the numbers, removing that ability is the point

each one is "blame.today but for X." pick the X. the protocol is the same; the surface changes. if blame works, those are the next vehicles.

why blame — and not some serious app?

because the surface is silly and the architecture is serious. if you ship this architecture inside an "elections reform" app, you spend three years on regulatory approval and four lawsuits. if you ship it inside "who's getting blamed today," you spend a weekend and find out whether the pattern works.

once it works, the serious applications get built by someone else on the same protocol. that's how patterns earn adoption — not by being announced, but by being demonstrated on something low-stakes nobody minds.
